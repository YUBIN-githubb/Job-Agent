# question-decoder 스킬

## 역할

자소서 문항 원문을 받아 다음 세 가지를 분석한다:

1. **표면 질문**: 문항이 글자 그대로 묻고 있는 것
2. **실제 평가 의도**: 출제자가 이 문항으로 확인하려는 것
3. **답변 전략**: 어떤 구조와 초점으로 답해야 하는가

**입력**: 자소서 문항 원문 + 카테고리 (data-parser가 부여한 것)
**출력**: 문항 해석 분석서

**이 스킬은 역량 태그를 부여하지 않는다.** 그건 competency-map의 일이다.
여기서는 "이 문항이 무엇을 평가하려는가"를 자연어로 분석한다.

## 분석 프레임워크

### 1. 문항 구조 분해

문항을 다음 요소로 분해한다:

**주제어**: 문항이 다루는 핵심 소재
- "도전적이었던 경험" → 주제어: 도전
- "지원 동기" → 주제어: 동기

**동사**: 문항이 요구하는 행동
- "서술하시오" → 경험 기반 서사
- "기술하시오" → 구체적 사실 나열
- "설명하시오" → 논리적 근거 제시

**범위 한정어**: 시간·대상·조건 제한
- "학창 시절" → 대학 이상 경험으로 한정
- "가장 ~한" → 최상급 하나만 선택 강제
- "최근 3년 내" → 시간 범위 명시
- "팀/조직에서" → 개인 경험 제외

**복합 질문 감지**: 한 문항 안에 여러 질문이 숨어 있는 경우
- "경험과 그로부터 배운 점" → (경험 서술) + (교훈 도출) 2개
- "강점과 약점" → (강점) + (약점) + (약점 극복 노력) 3개
- "지원 동기와 입사 후 포부" → (동기) + (포부) 2개

### 2. 평가 의도 추론

문항 카테고리별로 출제자가 실제로 확인하려는 것:

**경험서술 문항**
- 표면: "~한 경험을 서술하시오"
- 실제 평가: 과거 행동이 미래 행동을 예측할 수 있는가?
  - 구체적 상황에서 어떤 판단을 내렸는가 (의사결정 능력)
  - 행동이 결과로 이어지는 인과관계가 분명한가 (실행력)
  - 실패했다면 무엇을 학습했는가 (자기인식)

**지원동기 문항**
- 표면: "왜 우리 회사에 지원했는가"
- 실제 평가: 이 사람이 진지하게 우리 회사를 알아봤는가?
  - 회사에 대한 구체적 이해 (정보 수집 능력)
  - 본인 관심사와 회사 방향의 교차점 (적합성 판단)
  - "다른 회사가 아닌 이 회사인 이유" (차별화된 동기)

**직무적합성 문항**
- 표면: "본인이 이 직무에 적합한 이유"
- 실제 평가: 직무를 이해하고 있는가?
  - 직무 내용에 대한 정확한 이해 (JD 분석 능력)
  - 본인 역량과 직무 요구의 구체적 매칭 (자기 객관화)
  - 입사 후 기여 가능한 구체적 영역 (실현 가능성)

**강점약점 문항**
- 표면: "본인의 강점과 약점"
- 실제 평가: 자기 인식이 정확한가?
  - 강점을 직무 맥락에서 설명하는가 (추상적 자화자찬 배제)
  - 약점을 인지하고 대응하고 있는가 (성장 가능성)
  - 약점이 진짜 약점인가 (가짜 약점 감지: "일을 너무 열심히 합니다")

**성장과정 문항**
- 표면: "본인의 성장 과정"
- 실제 평가: 이 사람의 가치관은 무엇인가?
  - 어떤 경험이 현재의 관점을 형성했는가 (가치관의 근거)
  - 성장 방향이 직무/회사와 일치하는가 (문화 적합성)

**복합형 문항**
- 여러 카테고리가 섞인 경우, 각 하위 질문별로 위 분석을 적용
- 하위 질문 간 글자 수 배분 비율도 제안

### 3. 답변 전략 도출

분석 결과를 기반으로 답변의 구조적 전략을 제안한다:

**답변 구조 제안**: 문항 유형에 따른 최적 구조
- 경험서술 → 핵심주장-상황-행동-결과-의미 흐름
- 지원동기 → 회사이해-접점-기여포부 흐름
- 강점약점 → 강점(근거)-약점(인지-대응) 흐름

**초점 배분**: 어디에 글자 수를 가장 많이 써야 하는가
- 예: "도전 경험" 문항 → Action에 50%, Situation에 20%

**함정 경고**: 이 문항에서 흔히 빠지는 실수
- "도전 경험"에 성공담만 쓰는 것 (실패-극복이 더 효과적)
- "지원동기"에 회사 연혁을 나열하는 것 (소감이 아닌 접점 필요)
- "강점약점"에 가짜 약점을 쓰는 것 ("꼼꼼해서 느림" 류)

## 출력 형식

각 문항에 대해 다음 형식으로 반환한다:

```markdown
## 문항 [N] 분석

### 원문
> [문항 원문 그대로 — 한 글자도 수정 안 함]

### 카테고리
[data-parser가 부여한 카테고리] (변경 필요시 수정 카테고리 + 사유)

### 구조 분해
- 주제어: [핵심 소재]
- 동사: [요구 행동]
- 범위 한정어: [있다면]
- 하위 질문 수: [N개]
  - [하위질문 1]: [내용]
  - [하위질문 2]: [내용]

### 표면 질문
[문항이 글자 그대로 묻고 있는 것, 1~2문장]

### 실제 평가 의도
[출제자가 확인하려는 것, 2~3문장]
- 핵심 평가 축 1: [무엇을 보는가]
- 핵심 평가 축 2: [무엇을 보는가]
- (있다면) 핵심 평가 축 3

### 답변 전략
- 추천 구조: [구조명 + 간단 설명]
- 초점 배분: [어디에 비중을 둘 것인가]
- 글자 수 배분 (복합 문항 시): [하위질문별 비율]

### 함정 경고
- [흔한 실수 1]
- [흔한 실수 2]

### competency-map 전달 정보
- 평가 축 키워드: [핵심 평가 축에서 추출한 자연어 키워드]
- 문항 유형: [경험서술 / 지원동기 / 직무적합성 / 강점약점 / 성장과정 / 복합형]
- 회사 인재상 참조 필요: [있다면 company-report의 인재상 섹션]
```

## 특수 상황 처리

### 문항이 모호한 경우
"본인을 자유롭게 소개해 주세요" 같은 개방형 문항:
- 카테고리를 `복합형`으로 분류
- 평가 의도를 "지원자의 자기 인식 + 커뮤니케이션 능력"으로 해석
- 답변 전략에서 "직무 연관성 있는 경험 1개에 집중하라" 권장

### 직무 특화 문항
"OO 기술에 대한 이해도를 설명하시오" 같은 기술 문항:
- 카테고리를 `직무적합성`으로 분류
- 평가 의도에 "이론적 이해 + 실제 적용 경험" 모두 포함
- 답변 전략에서 "프로젝트 사례로 이해도를 증명하라" 권장

### data-parser의 카테고리가 틀린 경우
decoder가 분석해보니 카테고리가 맞지 않다면:
- 수정된 카테고리를 제안하고 사유를 명시
- 예: data-parser가 `경험서술`로 분류했지만 실제로는 `직무적합성`
  → "이 문항은 경험 자체가 아니라 직무와의 연결성을 묻고 있다"

### 글자 수가 극단적인 경우
- 200자 이하: "핵심 주장 한 문장 + 근거 한 문장"만 가능하다고 명시
- 1500자 이상: 복합 구조를 권장하고 하위 질문별 배분 제안

## 작업 순서

1. 문항 원문 + 카테고리 수령
2. 구조 분해 (주제어, 동사, 범위, 복합 질문 감지)
3. 평가 의도 추론 (카테고리별 프레임 적용)
4. 답변 전략 도출
5. 함정 경고 작성
6. competency-map 전달 정보 정리
7. 출력 형식으로 조립 후 반환

## 주의사항

- 문항 원문은 절대 수정하지 않는다 (따옴표 안에 그대로 보존).
- "이렇게 쓰세요" 같은 구체적 문장 생성은 하지 않는다 (essay-writer의 영역).
- 평가 의도 추론은 근거 기반이어야 한다. "아마 이걸 보려는 것 같다"가
  아니라 "이 유형의 문항은 일반적으로 ~를 평가한다"로 서술.
- company-report의 인재상이 있더라도, 이 스킬에서 인재상과 직접
  매핑하지 않는다. 그건 competency-map이 가중치로 반영할 일이다.
