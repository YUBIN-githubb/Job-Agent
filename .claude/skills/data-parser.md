# data-parser 스킬

## 역할

web-scraping이 넘긴 원시 본문에서 자소서 작성과 지원 관리에 필요한
필드를 추출하고, 저장 가능한 구조화 문서로 변환한다.

**입력**: web-scraping의 출력 (정제된 본문 + 메타 정보)
**출력**: YAML 프런트매터 + 마크다운 본문 형식의 구조화 문서

**이 스킬은 파일을 저장하지 않는다.** 저장은 jobs-db가 담당한다.

## 한국 채용공고의 표준 구조

한국 기업 공고는 대부분 다음 섹션들로 구성된다. 이 지식이 파싱 정확도의
핵심이다:

- **회사 소개**: 기업명, 사업 영역, 규모
- **직무/포지션**: "~엔지니어", "~매니저", "~사무직"
- **담당 업무**: 실제 수행할 일 (Job Description)
- **자격 요건 (필수)**: 학력, 경력, 기술 스택의 필수 조건
- **우대 사항**: 있으면 좋은 조건
- **근무 조건**: 급여, 복지, 근무지, 근무 형태
- **전형 절차**: 서류 → 인적성 → 면접 → 최종
- **접수 기간**: 시작일과 마감일
- **자기소개서 문항**: 있는 경우 개별 추출 (가장 중요)

## 추출 대상 필드

### 필수 필드 (누락 시 경고)
- `company`: 회사명 (정식 명칭)
- `position`: 직무명 (공고 제목 기준)
- `deadline`: 마감일 (YYYY-MM-DD, 상시모집이면 null)
- `employment_type`: 정규직 | 계약직 | 인턴 | 프리랜서
- `location`: 근무지 (도시 단위)

### 권장 필드
- `job_group`: 직군 (개발 / 기획 / 마케팅 / 영업 / 디자인 / 연구 등)
- `experience_level`: 신입 | 경력 | 신입+경력
- `requirements`: 필수 자격요건 배열
- `preferences`: 우대사항 배열
- `tech_stack`: 기술 스택 배열 (있는 경우)
- `salary`: 급여 (기재된 경우만)
- `essay_questions`: 자소서 문항 배열

### 메타 필드
- `id`: `{company-slug}-{position-slug}-{YYMM}` 형식
- `source_url`: 원본 URL (web-scraping이 전달)
- `collected_at`: 수집 일시 (ISO 8601)
- `raw_confidence`: web-scraping의 신뢰도 그대로 전달
- `status`: "수집완료" (초기 상태, jobs-db가 이후 변경)

## 자소서 문항 추출 (최우선 정확도)

자소서 문항은 find-experience, essay-draft의 핵심 입력이다.
다음 규칙을 엄격히 지킨다.

### 식별 패턴
다음 표현 뒤에 오는 질문들을 문항으로 인식한다:
- "자기소개서 항목", "자기소개서 문항"
- "1. ~", "2. ~" 번호 매겨진 질문 목록
- "문항 1", "Q1", "[1]" 같은 표기
- "다음 항목에 대해 작성해 주세요" 이후 목록
- "아래 내용을 기술해 주세요" 이후 질문들

### 글자 수 제한 파싱
문항 뒤에 자주 나오는 표현:
- "(700자 이내)" → char_limit: 700
- "(500자 내외)" → char_limit: 500
- "(최소 300자 ~ 최대 800자)" → char_limit: 800 (최대값 기준)
- "(공백 포함 1000자)" → char_limit: 1000, char_limit_note: "공백 포함"
- "(공백 제외)" → char_limit_note: "공백 제외"
- 제한 없음 → char_limit: null

한국 자소서는 공백 포함이 일반적이므로, 명시되지 않으면 "공백 포함"으로 기본값 설정.

### 문항별 추출 형식

각 문항마다 다음을 기록한다:

```yaml
essay_questions:
  - number: 1
    question: "문항 원문 전체"
    char_limit: 700
    char_limit_note: "공백 포함"
    category: 지원동기
```

### 카테고리 판별

문항 본문의 키워드로 카테고리를 추정한다:

| 키워드 포함 | 카테고리 |
|---|---|
| "지원 동기", "왜 우리 회사", "입사 후 포부" | `지원동기` |
| "경험", "사례", "~한 적이 있다" | `경험서술` |
| "강점", "장점", "약점", "단점" | `강점약점` |
| "성장 과정", "가치관", "인생에서" | `성장과정` |
| "직무 역량", "기술적 준비", "전문성" | `직무적합성` |
| 여러 질문이 섞인 경우 | `복합형` |

카테고리는 find-experience가 에피소드 후보를 필터링할 때 사용된다.
판별이 애매한 경우 `복합형`으로 분류한다.

## 정규화 규칙

### 회사명
- `㈜삼성전자`, `삼성전자(주)`, `SAMSUNG ELECTRONICS CO., LTD.` → `삼성전자`
- 그룹사 소속이면 정식 명칭 유지 (`현대모비스`를 `현대`로 축약 금지)
- 스타트업의 영문명은 원본 유지 (`Toss`, `당근`)

### 마감일 파싱
다양한 표현을 YYYY-MM-DD로 변환:
- `2026년 5월 15일(금) 23:59` → `2026-05-15`
- `~5/15` → 현재 연도 기준 `2026-05-15`
- `05/15까지` → `2026-05-15`
- `상시모집`, `수시채용` → `null`
- `채용 시 마감` → `null` + 본문에 "채용 시 마감" 주석

### ID 생성
파일명과 id가 동일해야 한다:
- 회사 슬러그: 공백/특수문자 제거, 소문자화, 한글은 로마자 변환
  - `삼성전자` → `samsung-electronics`
  - `당근마켓` → `daangn` (또는 `karrot`)
- 직무 슬러그: 동일 규칙
  - `반도체 설계 엔지니어` → `semi-designer`
- YYMM: 수집 시점 연월 각 2자리
  - 2026년 4월 → `2604`
- 예: `samsung-electronics-semi-designer-2604`

## 특수 상황 처리

### 복수 직무가 섞인 공고
"개발팀 신입 채용 - 백엔드/프론트/데브옵스"처럼 여러 직무가 섞인 경우:
- 사용자에게 "어느 직무로 파싱할지" 선택 요청
- 또는 각 직무별로 별도 레코드 생성 (사용자 확인 후)

### 필수 필드 누락
`company`, `position`, `deadline` 중 하나라도 추출 실패 시:
- 추출한 부분만으로 임시 저장하지 않는다
- 사용자에게 누락 필드를 명시하고 수동 입력 요청
- 원시 텍스트는 jobs-db가 `data/jobs/raw/`에 보관하도록 넘긴다

### 자소서 문항이 본문엔 있는데 못 찾은 경우
원시 본문에서 "자기소개서", "자소서", "항목" 키워드로 재검색한다.
그래도 못 찾으면:
- `essay_questions: []`로 빈 배열 반환
- 본문 하단 `## 파싱 노트`에 "자소서 문항 섹션 식별 실패" 명시
- 사용자에게 "이 공고에 자소서 문항이 있다면 직접 알려주세요" 안내

### 자소서 문항이 아예 없는 공고
서류만 접수하는 경우(기술 직군에서 흔함):
- `essay_questions: []`
- `## 파싱 노트`에 "자소서 문항 없음, 포트폴리오/이력서 중심 전형" 명시

## 작업 순서

1. web-scraping 출력의 메타 정보 읽기 (신뢰도, 수집 방식, 원본 URL)
2. 신뢰도가 `low`면 파싱 진행 여부를 사용자에게 먼저 확인
3. 본문에서 섹션 식별 (제목/볼드 기준)
4. 필수 필드 추출 (없으면 사용자 확인)
5. 권장 필드 추출
6. 자소서 문항 추출 (최우선 정확도)
7. 정규화 규칙 적용 (회사명, 마감일, ID)
8. YAML 프런트매터 + 마크다운 본문으로 조립
9. jobs-db에 전달 가능한 상태로 반환

## 주의사항

- 이 스킬은 **파싱만** 한다. 파일 쓰기, 인덱스 업데이트, 중복 체크는
  jobs-db의 책임이다.
- 원문에 없는 내용을 추론해서 채우지 않는다 (예: 언급 없는 기술 스택
  임의 추가 금지).
- 자소서 문항 원문은 **한 글자도 바꾸지 않는다** (조사, 존댓말 포함).
  다음 파이프라인에서 원문 그대로 필요.
