# tracker 스킬

## 역할

파일 시스템에 흩어진 공고·리포트·자소서 파일들을 스캔해
통합 대시보드를 만들고, 상태 전이·알림·리포트를 처리한다.

**입력**: apply-manager의 요청 (현황 조회 / 상태 변경 / 리포트 생성)
**출력**: 대시보드 파일, 주간 리포트 파일, 상태 전이 결과

## 작업 영역

```
data/tracking/
├── dashboard.md                 # 통합 대시보드 (자동 갱신)
├── weekly-{YYYY-MM-DD}.md       # 주간 리포트
└── alerts.md                    # 활성 알림 목록
```

## 기능 1: 대시보드 생성/갱신

### 데이터 수집 절차
1. `data/jobs/collected/*.md` Glob → 전체 공고 목록
2. 각 파일의 프런트매터만 빠르게 Read:
   - `id`, `company`, `position`, `status`, `deadline`
   - `linked_files` (어디까지 진행됐는지)
3. 현재 날짜 기준 D-day 계산
4. 상태별 그룹화 및 마감순 정렬

### 대시보드 형식

```markdown
---
last_updated: 2026-04-16T10:30:00+09:00
total: 5
by_status:
  수집완료: 1
  분석완료: 1
  자소서작성중: 1
  자소서완료: 1
  지원완료: 1
  마감: 0
---

# 취업 준비 대시보드

> 마지막 갱신: 2026-04-16 10:30
> 총 공고: 5건 | 진행중: 4건 | 지원완료: 1건

## 🔴 긴급 (D-3 이내)
| 회사 | 직무 | 마감 | 상태 | 다음 행동 |
|---|---|---|---|---|
| SK하이닉스 | 메모리 설계 | D-2 | 자소서작성중 | 자소서 완성 시급 |

## ⚠️ 주의 (D-7 이내)
| 회사 | 직무 | 마감 | 상태 | 다음 행동 |
|---|---|---|---|---|
| 네이버 | 백엔드 | D-5 | 분석완료 | 문항 분석 필요 |

## 📋 진행중
| 회사 | 직무 | 마감 | 상태 | 다음 행동 |
|---|---|---|---|---|
| 삼성전자 | 반도체 설계 | D-29 | 자소서완료 | 지원 제출 |
| 당근 | PM | 상시 | 수집완료 | 기업 분석 필요 |

## ✅ 완료
| 회사 | 직무 | 제출일 | 결과 |
|---|---|---|---|
| LG CNS | 데이터 | 2026-04-10 | 결과대기 |

## 📊 통계
- 이번 달 지원: 2건
- 전체 합격률: - (결과 대기 중)
- 가장 많이 사용한 에피소드: capstone-leadership (3회)
```

### "다음 행동" 판정 로직

상태별로 다음 행동을 자동 결정한다:

| 현재 상태 | 다음 행동 | 제안 에이전트 |
|---|---|---|
| `수집완료` | 기업 분석 필요 | @company-analyst |
| `분석완료` | 문항 분석 필요 | @question-analyst |
| `문항분석완료` | 자소서 작성 시작 | @essay-writer |
| `자소서작성중` | 자소서 완성 필요 | @essay-writer |
| `자소서완료` | 지원 제출 | (사용자 직접) |
| `지원완료` | 결과 대기 | - |
| `결과대기` | 결과 입력 대기 | (사용자 직접) |

마감 D-3 이내이면 다음 행동 앞에 "시급" 표시 추가.

## 기능 2: 상태 전이

### 전이 실행 절차
1. 대상 공고 파일 Read
2. 현재 status 확인
3. 요청된 전이가 유효한지 검증 (아래 전이표 참조)
4. 프런트매터의 `status` 갱신
5. `status_history` 배열에 이력 추가
6. 파일 재쓰기
7. 대시보드 갱신

### 유효한 전이표

```
수집완료 → 분석완료, 마감
분석완료 → 문항분석완료, 마감
문항분석완료 → 자소서작성중, 마감
자소서작성중 → 자소서완료, 마감
자소서완료 → 지원완료, 마감
지원완료 → 결과대기
결과대기 → 최종결과
```

역방향 전이는 허용하지 않는다. 사용자가 요청하면:
"상태를 되돌리는 것은 지원하지 않습니다. 새 공고로 다시 시작해 주세요."

### 자동 전이

**linked_files 기반 자동 승격**:
대시보드 갱신 시 각 공고의 status와 linked_files를 비교한다.
linked_files가 status보다 상위 단계를 가리키면 자동 승격:

```python
# 판정 로직 (개념 의사코드)
if "essays_final" in linked_files and status < "자소서완료":
    status = "자소서완료"
elif "essays" in linked_files and status < "자소서작성중":
    status = "자소서작성중"
elif "question_analysis" in linked_files and status < "문항분석완료":
    status = "문항분석완료"
elif "company_report" in linked_files and status < "분석완료":
    status = "분석완료"
```

자동 승격 시 status_history에 `auto_promoted: true` 플래그.

**마감 자동 전이**:
deadline이 지난 공고 중 status가 `지원완료`/`결과대기`/`최종결과`가
아닌 건은 자동으로 `마감`으로 전이.

### status_history 기록 형식

```yaml
status_history:
  - status: 수집완료
    date: 2026-04-10T10:00:00+09:00
    agent: job-scraper
  - status: 분석완료
    date: 2026-04-10T14:20:00+09:00
    agent: company-analyst
    auto_promoted: true
  - status: 지원완료
    date: 2026-04-20T09:00:00+09:00
    agent: apply-manager
    note: 사용자 확인
```

## 기능 3: 마감 알림

### 알림 생성 로직
대시보드 갱신 시 함께 실행:

1. 전체 공고 중 `status`가 `지원완료`/`결과대기`/`최종결과`/`마감`이
   아닌 건만 대상
2. 현재 날짜 기준 D-day 계산
3. D-7 / D-3 / D-1 기준으로 알림 생성
4. `data/tracking/alerts.md` 갱신

### 알림 파일 형식

```markdown
---
generated_at: 2026-04-16T10:30:00+09:00
active_alerts: 2
---

# 활성 알림

## 🚨 D-1 이내
(없음)

## 🔴 D-3 이내
- **SK하이닉스 - 메모리 설계** (D-2, 2026-04-18)
  - 현재 상태: 자소서작성중
  - 필요 행동: 자소서 완성 후 즉시 제출
  - 호출: @essay-writer

## ⚠️ D-7 이내
- **네이버 - 백엔드** (D-5, 2026-04-21)
  - 현재 상태: 분석완료
  - 필요 행동: 문항 분석 → 자소서 작성 → 제출
  - 호출: @question-analyst
```

### 알림 자동 해제
- 해당 공고가 `지원완료` 이상으로 전이되면 알림 자동 제거
- 마감일이 지나면 알림 대신 `마감` 상태 전이

## 기능 4: 주간 리포트

### 생성 트리거
- 사용자 명시 요청: "주간 리포트 만들어줘"
- 매주 월요일 첫 호출 시 자동 제안

### 리포트 내용

```markdown
---
period: 2026-04-10 ~ 2026-04-16
generated_at: 2026-04-16T10:30:00+09:00
---

# 주간 리포트 (4/10 ~ 4/16)

## 이번 주 변동
- [신규] 삼성전자 반도체 설계 공고 수집 (4/10)
- [진행] 삼성전자 기업 분석 완료 (4/10)
- [진행] 삼성전자 문항 분석 완료 (4/12)
- [진행] 삼성전자 자소서 초안 작성 (4/14)
- [완료] LG CNS 데이터 엔지니어 제출 완료 (4/10)

## 다음 주 할 일 (우선순위순)
1. 🔴 SK하이닉스 자소서 완성 (D-2, 4/18 마감)
2. ⚠️ 네이버 문항 분석 시작 (D-5, 4/21 마감)
3. 삼성전자 자소서 퇴고 및 확정
4. 당근 기업 분석 시작 (상시, 급하지 않음)

## 통계
- 활성 공고: 4건
- 이번 주 완료: 1건 (LG CNS 제출)
- 자소서 진행률: 삼성 80%, SK 40%, 네이버 0%
- 에피소드 사용 현황: capstone-leadership(3), picket-problem-solving(2)

## 에피소드 커버리지 경고
- '윤리의식' 태그 에피소드 0개 (관련 문항 발견 시 대비 필요)
- capstone-leadership 3회 사용 — 다른 에피소드 활용 권장
```

### 변동 감지 방식
- 이전 주간 리포트의 `generated_at`과 현재 사이의 변동을 검출
- 공고 파일의 `status_history` 타임스탬프 기준
- 이전 리포트가 없으면 전체 공고를 "신규"로 처리

## 기능 5: 에피소드 사용 현황

### 추적 대상
`data/experiences/episodes/*.md`의 프런트매터에서:
- `used_count`: 총 사용 횟수
- `last_used`: 마지막 사용 회사/시기

### 경고 기준
- 동일 에피소드 3회 이상 사용 → "다양성 부족" 경고
- 특정 카테고리 에피소드만 집중 사용 → "역량 편중" 경고
- 대시보드의 통계 섹션에 포함

## 특수 상황 처리

### 공고 파일이 손상/누락된 경우
- 인덱스에는 있는데 파일이 없는 경우 → "파일 누락" 경고
- 프런트매터가 파싱 안 되는 경우 → 해당 건 "확인 필요"로 표시

### data/tracking/ 디렉토리가 없는 경우
- 첫 실행 시 자동 생성
- 빈 대시보드 초기화

### 동시에 10개 이상 공고 진행
- 대시보드가 길어지므로 "긴급/주의" 섹션만 요약 출력
- 전체 목록은 파일로 저장하고 링크 제공

## 작업 순서

### 대시보드 갱신
1. `data/jobs/collected/*.md` 전체 Glob
2. 각 파일 프런트매터 읽기
3. linked_files 기반 자동 승격 체크
4. 마감 자동 전이 체크
5. 상태별 그룹화 + 마감순 정렬
6. 알림 판정
7. 다음 행동 결정
8. `data/tracking/dashboard.md` 쓰기
9. `data/tracking/alerts.md` 쓰기

### 상태 변경
1. 대상 공고 특정
2. 현재 status 확인
3. 전이 유효성 검증
4. 프런트매터 갱신 + 이력 추가
5. 파일 쓰기
6. 대시보드 갱신

### 주간 리포트
1. 전체 공고 스캔
2. 이전 리포트 대비 변동 검출
3. 다음 주 할 일 우선순위 정렬
4. 에피소드 사용 현황 집계
5. `data/tracking/weekly-{date}.md` 쓰기

## 주의사항

- 다른 에이전트의 산출물을 직접 수정하지 않는다
  (예외: 공고 파일의 `status`와 `status_history`만 갱신 가능).
- 대시보드는 파일 시스템의 현재 상태를 반영하는 "스냅샷"이다.
  캐시하지 않고 매번 실제 파일을 스캔한다.
- 알림은 "제안"이다. 사용자에게 강제하지 않는다.
- 주간 리포트는 마지막 리포트 이후의 변동만 다룬다.
  전체 히스토리가 필요하면 대시보드를 참조하도록 안내.
