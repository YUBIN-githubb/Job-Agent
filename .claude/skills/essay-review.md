# essay-review: 자소서 초안 퇴고 & 개선 제안

## 역할

완성된 초안(또는 기존 자소서)을 받아 세 가지 관점에서 검토하고,
각 항목에 대한 구체적 개선안을 번호 매긴 형태로 제안한다.

**이 스킬은 원문을 덮어쓰지 않는다.** 제안만 하고 최종 결정은 사용자의 몫이다.
"자동 퇴고"는 자소서의 주도권을 사용자에게서 빼앗는 행위이므로 하지 않는다.

## 입력 형식

다음 중 하나의 형태로 입력을 받는다:

1. **파일 경로**: `data/essays/xxx.md` 같은 기존 초안 파일
2. **직접 붙여넣기**: 사용자가 채팅에 자소서 텍스트를 직접 입력
3. **문항 + 본문 쌍**: 문항과 답변 텍스트가 함께 주어진 경우

문항이 입력에 포함되지 않았다면 먼저 "어떤 문항에 대한 답변인가요?"
라고 확인한다. 문항 없이 검토하면 "문항 충실성"을 판단할 수 없기 때문이다.

## 검토 3축

### 1축: 문항 충실성 (Relevance)
**질문에 정확히 답했는가.**

체크 포인트:
- 문항의 모든 하위 질문에 답했는가 (복합 문항 누락 체크)
- 첫 문장이 문항의 핵심에 직접 닿는가
- 에피소드가 문항의 동사(이끌다/해결하다/설득하다)와 일치하는가
- 결론이 다시 문항으로 돌아와 마무리되는가

가장 흔한 실패는 "에피소드 자체는 흥미로운데 문항과 미묘하게 어긋나는"
경우다. 이걸 먼저 걸러낸다.

### 2축: 구체성 (Specificity)
**누구나 쓸 수 있는 글이 아닌가.**

체크 포인트:
- 숫자가 들어있는가 (팀 규모, 기간, 정량 성과)
- 고유명사가 들어있는가 (프로젝트명, 도구명, 회사/기관명)
- 동사가 구체적인가 ("노력했다" → "주 2회 스프린트 회고를 도입했다")
- "나만이 쓸 수 있는 디테일"이 최소 2개 있는가

디테일이 부족한 문장을 "누구나형 문장"으로 분류하고,
에피소드 원본에 더 구체적 재료가 있는지 찾아 교체 제안을 한다.

### 3축: 차별성 (Distinctiveness)
**클리셰에 빠지지 않았는가.**

금지/경계 표현 검출:
- 자기 선언형: "저는 ~한 사람입니다", "~를 가치관으로 삼고 있습니다"
- 추상 부사: "열정을 다해", "최선을 다해", "도전정신을 가지고"
- 모호 어미: "~같습니다", "~인 것 같습니다", "~라고 생각됩니다"
- 상투적 연결: "귀사의 인재상에 부합", "귀사의 일원이 되어"
- 명언/속담 인용으로 시작
- "~한 경험을 통해 ~를 배웠다"의 형식적 마무리

각 표현 검출 시 원문 인용 + 교체 문장 후보 2개 제시.

## 출력 형식

```markdown
# 자소서 검토 결과

## 검토 대상
- 문항: [문항 원문 또는 키워드]
- 글자 수: [현재 XXX자 / 제한 XXX자] [상태: OK / 초과 / 미달]
- 검토 시각: [ISO 날짜]

## 종합 진단
[3~4문장. 이 초안의 전반적 품질 평가 + 가장 시급한 개선 방향]

## 치명적 이슈 ([N]개)
> 수정하지 않으면 자소서의 설득력이 크게 훼손되는 항목.

### [1] 문항 이탈
- 문제: "문항은 '리더십'을 묻는데 답변은 '문제해결'에 치우침"
- 원문 인용: "[본문 일부]"
- 제안: "[구체적 교체 문장 또는 구조 변경안]"

### [2] ...

## 권장 개선 ([N]개)
> 수정하면 눈에 띄게 좋아지는 항목.

### [1] 구체성 부족
- 위치: "첫 문단 2번째 문장"
- 원문: "많은 노력을 기울였다"
- 제안 A: "주 20시간씩 3개월간 투자했다"
- 제안 B: "출근 전 2시간을 고정 학습 시간으로 확보했다"
- 사유: 숫자 + 구체적 행동으로 디테일 복원

### [2] 클리셰 표현
- 원문: "열정을 다해 임했다"
- 제안 A: "마감 2주 전부터 매일 21시까지 자리에 남았다"
- 제안 B: (삭제하고 다음 문장으로 바로 연결)
- 사유: 추상 부사는 구체적 행동으로 치환하거나 삭제

## 글자 수 조정 제안 ([초과/미달 시만])
- 현재 820자, 제한 800자 → 20자 감축 필요
- 감축 후보 구간: [특정 문장 인용]
- 대안: 다음 표현으로 교체 시 15자 절감 → [교체안]

## 선택적 제안 ([N]개)
> 취향에 따라 고려할 수 있는 개선.

### [1] 첫 문장 대안
- 현재: "[원문]"
- 대안 A (주장 강조형): "[문장]"
- 대안 B (질문 도입형): "[문장]"

## 다음 단계
1. 치명적 이슈부터 검토해 원문 수정
2. 수정 후 재검토를 원하면 "다시 검토해줘" 요청
3. 확정되면 최종본을 `data/essays/{파일명}-final.md`로 저장
```

## 검토 원칙

### 1. 정량적 근거 우선
"글이 좀 밋밋해요" 같은 인상 비평은 하지 않는다.
"2문단에 숫자가 0개, 고유명사가 0개"처럼 셀 수 있는 근거로 말한다.

### 2. 대안 없는 비판 금지
"이 문장은 별로다"로 끝내지 않는다.
모든 치명적/권장 이슈는 반드시 교체안 1개 이상을 제시한다.
교체안이 떠오르지 않으면 "이 부분은 에피소드 원본에서 다른 재료를
가져와야 함"처럼 다음 행동을 명시한다.

### 3. 우선순위를 엄격히 지킨다
- **치명적**: 문항 이탈, 사실관계 오류, 법적/윤리적 위험 표현
- **권장**: 구체성 부족, 클리셰, 어색한 연결
- **선택적**: 문체 취향, 첫 문장 스타일 대안

치명적이 너무 많다(5개 이상)면 이 초안은 에피소드 재선정이 필요하다는
신호다. find-experience를 재실행하도록 안내한다.

### 4. 피드백 누적을 고려한다
같은 초안을 두 번째로 검토하는 경우,
이전 검토에서 지적한 항목이 반영되었는지 먼저 확인한다.
이미 반영된 항목은 제외하고 새로운 이슈에만 집중한다.
이전 검토 이력은 `data/essays/{파일명}.review.md`로 저장한다.

### 5. 한국 자소서 특유의 맥락을 인지한다
- 대기업 공채: 형식적 완결성과 직무 연관성이 강하게 평가됨
- 스타트업: 구체적 실행력과 본인만의 시각이 더 중요
- 공공기관: 직무기술서 키워드와의 정확한 매칭이 결정적
- 외국계: 결과 중심 서술과 수치 증명이 필수

회사 분석 리포트가 `data/reports/`에 있다면 해당 맥락을 가중치로 반영한다.

## 특수 상황 처리

### 초안이 문항 범위를 이탈한 경우
에피소드 자체의 방향이 틀린 것이므로 부분 수정이 아닌
재작성이 필요하다고 판단한다. 이때는 개선안 나열 대신
"이 에피소드는 이 문항에 맞지 않습니다. find-experience로
다른 에피소드를 찾으세요"라고 안내한다.

### 같은 회사 자소서 내 중복 검토
같은 회사에 제출할 자소서 여러 문항을 연달아 검토하는 경우,
문항 간 톤/소재 중복을 추가로 체크한다.
"1번 문항과 3번 문항이 모두 캡스톤 프로젝트를 소재로 사용 중.
3번 문항은 다른 에피소드로 교체 권장"처럼 회사 단위 점검을 한다.

### 글자 수 대폭 초과 (110% 이상)
essay-draft에서 조절 실패한 경우다. 전면 재작성이 필요하므로
개선 제안 대신 "어느 부분을 살리고 어느 부분을 포기할지"를
사용자에게 먼저 묻는다.

### 지원동기 문항 검토
경험 서술형과 평가 축이 다르다.
- 회사에 대한 구체적 이해가 드러나는가 (뉴스/제품/전략 언급)
- 본인의 관심과 회사 방향의 접점이 설득력 있는가
- "왜 다른 회사가 아닌 이 회사인가"에 답했는가

이 세 축으로 별도 검토한다.

## 작성 워크플로

1. 입력 검증 (초안 + 문항 둘 다 있는가)
2. 글자 수 측정
3. 문항 유형 판별
4. 3축 검토 실행
5. 우선순위별로 이슈 분류
6. 각 이슈에 대한 교체안 작성
7. 출력 형식에 맞춰 정리
8. `data/essays/{파일명}.review.md`에 검토 이력 저장
9. 사용자 수정 후 재검토 요청 대기

## 사용 예시

입력: `data/essays/미지정-도전경험.md` 검토 요청

동작:
1. 파일 Read
2. 파일 상단의 문항 메타데이터 확인
3. 본문 글자 수 측정 → 582자 (제한 600자 내 OK)
4. 3축 검토:
   - 문항 충실성: "도전" 동사와 에피소드 Action 일치 여부 체크
   - 구체성: 숫자/고유명사 카운트
   - 차별성: 금지 표현 검출
5. 검토 결과 출력
6. `data/essays/미지정-도전경험.review.md` 저장

## 검토 이력 파일 형식

`data/essays/{파일명}.review.md`:

```yaml
---
target: 미지정-도전경험.md
last_reviewed: 2026-04-16
review_count: 1
---

# 1차 검토 (2026-04-16)
- 치명적 이슈 2건 지적
- 권장 개선 4건 지적
- 사용자 대응: [확인 후 업데이트]
```

이 파일은 2차 검토 시 중복 제안을 방지하는 용도로 사용된다.
