# essay-draft: 자소서 초안 작성

## 역할

선정된 에피소드, 자소서 문항, 제약 조건(글자 수, 회사 맥락)을 입력받아
첫 번째 초안을 작성한다. 완벽한 글이 아닌 "검토 가능한 초안"을 목표로 한다.

**이 스킬은 퇴고하지 않는다.** 표현 다듬기와 차별화는 essay-review가 담당한다.
여기서는 구조가 무너지지 않은 초안을 빠르게 만드는 것이 우선이다.

## 입력 형식

이 스킬은 다음을 입력으로 받는다:

1. **문항 원문** (필수): 자소서 질문 전체 텍스트
2. **글자 수 제한** (필수): 공백 포함/미포함 명시
3. **선정 에피소드 ID** (필수): find-experience가 반환한 1순위
4. **회사/직무 정보** (선택): 분석 리포트 경로 또는 핵심 요약
5. **톤 지정** (선택): 기본은 "담담하고 구체적인 서술체"

입력이 불완전하면 먼저 find-experience를 호출해 에피소드를 매칭하도록 안내한다.

## 데이터 경로

- 에피소드 본문: `data/experiences/episodes/{id}.md`
- 프로젝트 배경: `data/experiences/projects/{source}.md` (있다면)
- 회사 분석: `data/reports/{company}.md` (있다면)
- 초안 출력: `data/essays/{company}-{question-key}.md`

## 작성 원칙

### 1. 질문에 먼저 답한다
첫 문장이 문항의 핵심 질문에 직접 답해야 한다.
"저는 ~한 경험이 있습니다"로 시작하지 말고,
"~이 가능했던 이유는 ~였다"처럼 주장이 먼저 오는 문장을 쓴다.

### 2. STAR를 그대로 옮기지 않는다
에피소드 파일의 S/T/A/R은 재료이지 최종 구조가 아니다.
자소서에서는 보통 다음 흐름이 자연스럽다:

```
[핵심 주장] → [상황의 긴장감] → [행동의 디테일] → [결과와 의미]
```

Situation과 Task는 압축해서 2~3문장으로, Action은 구체적 디테일을
살려 자세히, Result는 수치로 마무리하되 "이 경험이 나에게 남긴 것"을
한 줄 덧붙이는 게 좋다.

### 3. 수치와 고유명사를 보존한다
에피소드 본문의 구체적 숫자(정확도 94.2%, 팀원 5명)와
고유명사(OO전자, 파이썬)는 자소서에도 반드시 옮긴다.
이것들이 초안을 "누구나 쓸 수 있는 글"에서 분리시키는 장치다.

### 4. 회사 연결은 마지막 문단에서
회사 분석 리포트가 주어졌다면, 마지막 1~2문장에서
"이 경험이 [회사명]의 [구체적 방향성]에 어떻게 기여할 수 있는가"로
자연스럽게 연결한다. 회사명을 억지로 중간에 끼워넣지 않는다.

### 5. 금지 표현
- "저는 ~한 사람입니다" (자기 선언형)
- "도전정신을 가지고", "열정을 다해" (추상 부사)
- "~같습니다", "~인 것 같습니다" (모호한 어미)
- "귀사", "OO사의 인재상에 부합하여" (상투적 연결)
- 첫 문장의 명언/속담 인용

## 글자 수 관리

### 목표 길이 계산
- 제한이 "800자 이내"라면 → 740~780자 목표
- 제한이 "500자 내외"라면 → 480~520자 목표
- 여유를 5% 두어 essay-review의 다듬기 공간을 남긴다

### 공백 포함/미포함
한국 자소서는 공백 포함 기준이 일반적이다.
명시되지 않은 경우 공백 포함으로 계산하고, 초안 끝에
실제 글자 수를 `[XXX자 / 공백 포함]` 형식으로 표기한다.

### 초과 시 대응
- 110% 이내: 경고만 하고 초안 제공 (review에서 다듬기 가능)
- 110% 초과: Action 부분에서 과감히 압축한 재작성
- 70% 미만: 원인이 에피소드 빈약인지 검토 요청

## 출력 형식

```markdown
# [회사명] - [문항 키워드] 초안

## 문항
> [원문 그대로]

## 선정 에피소드
- ID: [episode-id]
- 핵심 메시지: [에피소드의 주장]

## 초안

[본문 — 한 덩어리로 작성, 불필요한 줄바꿈 없이]

---
- 글자 수: [XXX자 / 공백 포함]
- 제한: [XXX자 이내]
- 다음 단계: essay-review 스킬로 퇴고 요청
```

## 특수 상황 처리

### 에피소드가 문항에 일부만 맞을 때
find-experience가 "적합도 중간"으로 표시한 경우,
에피소드 전체를 쓰기보다 문항과 맞는 부분만 발췌해 재구성한다.
이 경우 초안 하단에 "어떤 측면을 선택적으로 활용했는지" 주석을 단다.

### 문항이 여러 질문을 포함할 때
"본인의 강점과 약점을 서술하고 ~"처럼 복합 문항인 경우,
각 질문별로 글자 수를 할당하고 단락을 분리한다.
약점 질문에서는 단순한 약점 고백이 아니라 "인지와 대응"을 보여준다.

### 지원동기 문항
에피소드 기반 서술이 아닌 "회사 분석 + 본인 관심사 접점" 구조로 전환한다.
이 경우 find-experience 결과 대신 회사 분석 리포트를 주 재료로 쓴다.

### 글자 수가 200자 이하일 때
압축 서술이 어려우므로 에피소드 1개의 Action 한 가지에만 집중한다.
"무엇을 했는지"와 "결과"만 남기고 상황 설명은 최소화한다.

## 작성 워크플로

1. 입력 검증 (문항, 글자 수, 에피소드 ID가 모두 있는가)
2. 에피소드 본문 Read
3. (있다면) 프로젝트 배경 Read
4. (있다면) 회사 분석 리포트 Read
5. 문항 유형 판별 (경험 서술형 / 지원동기형 / 복합형)
6. 초안 작성 (목표 글자 수에 맞춰)
7. 글자 수 계산 및 출력 형식으로 정리
8. `data/essays/` 아래에 파일로 저장
9. essay-review로의 다음 단계 안내

## 사용 예시

입력:
- 문항: "학창 시절 가장 도전적이었던 경험과 그로부터 배운 점"
- 글자 수: 600자 이내
- 에피소드: picket-problem-solving
- 회사: 없음

동작:
1. `data/experiences/episodes/picket-problem-solving.md` Read
2. 문항 유형 = "경험 서술형"
3. 핵심 주장을 첫 문장에, 상황 압축, Action 강조, Result로 마무리
4. 560~580자 목표로 작성
5. `data/essays/미지정-도전경험.md` 저장
6. "essay-review로 다듬기 요청하세요" 안내
