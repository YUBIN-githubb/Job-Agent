# Lock 범위와 트랜잭션 범위 불일치로 인한 동시성 문제 해결

## Situation
공연 예매 시 예약된 좌석 수를 업데이트하는 과정에서 Redisson 분산 Lock을 적용했음에도 동시 요청 시 카운트가 정상 반영되지 않는 Race Condition 지속 발생.
원인 분석 결과 'Lock 해제'가 '트랜잭션 커밋'보다 먼저 일어나는 현상 확인.
트랜잭션 커밋으로 DB에 반영되기 전 다른 스레드가 Lock을 획득하여 갱신 전 데이터를 읽어가는 것이 원인임을 파악.

## Task
트랜잭션 커밋 완료 후 Lock이 해제되도록 Lock 범위를 트랜잭션 범위보다 넓게 확장하여 데이터 정합성 보장 필요.

## Action
1. 별도 트랜잭션 분리 방식은 예매 로직 실패 시 롤백 구현 복잡도 증가로 배제.
2. DB 비관적 Lock은 커넥션 점유 시간 증가에 따른 전체 처리량 저하 우려로 배제.
3. Redis에 예약 좌석 수 데이터를 관리하는 방식은 DB-Redis 간 동기화 문제와 시스템 복잡도 증가로 배제.
결론적으로 Facade 패턴 도입. 
비즈니스 로직을 감싸는 Facade 클래스를 구현하여 Facade 계층에서 Lock 획득 → Service 트랜잭션 메서드 호출 → 트랜잭션 커밋 완료 후 finally 블록에서 Lock 해제하는 구조로 변경.

## Result
Lock 생명주기가 트랜잭션 생명주기를 포함하는 구조로 개선하여 DB 커밋 시점 차이로 인한 동시성 이슈 해결.
비즈니스 로직과 Lock 제어 로직 분리로 코드 가독성 및 유지보수성 확보.

## 이 에피소드의 핵심 메시지
"성능 저하와 시스템 복잡도 등 다양한 대안의 트레이드오프를 논리적으로 분석. facade 패턴 기반의 동시성 제어 구조를 구현한 경험"
