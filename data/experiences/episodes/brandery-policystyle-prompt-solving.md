# OCR → 전처리 → 검수 3단계 프로세스에서 정책 기반 단일 프롬프트 방식으로 전환

## Situation
초기 설계는 GPT로 OCR 작업→ 미션 타입별 전처리 프롬프트로 필요 텍스트 추출→ 코드로 DB 값과 비교하는 3단계 구조.
미션 타입이 추가될 때마다 전처리 프롬프트 + 검수 함수를 각각 작성해야 하는 구조.
플랫폼 8개 × 미션 타입 8개 조합으로 검수 함수 관리 복잡도가 높아지는 상황.
LLM을 OCR 용도로만 사용하고 있어 GPT의 추론 능력을 활용하지 못하는 구조.

## Action
배달의민족 기술블로그의 "GPT 기반 메뉴 이미지 검수" 사례를 참고하여 정책 기반 검수 방식으로 방향 전환.
검수 기준을 자연어 정책으로 정의하고 플랫폼별 × 미션 타입별 필요 정책을 매핑.
GPT가 OCR부터 정책 판단까지 하나의 프롬프트로 수행하도록 구현.

## Result
미션 추가 시 필요한 작업이 2가지(전처리 프롬프트 + 검수 함수)에서 1가지(정책 추가)로 감소.
검수 로직을 코드가 아닌 자연어 정책으로 관리하여 비개발자도 정책 내용 파악 가능.
GPT의 추론 능력을 활용해 단순 텍스트 매칭이 아닌 맥락 기반 검수 가능.

## 이 에피소드의 핵심 메시지
"검수 로직을 하드코딩된 함수가 아닌 자연어 정책으로 전환함으로써, 다수의 플랫폼과 미션 확장에 유연하게 대응가능 하도록 한 경험. 비개발자와의 협업 효율까지 고려한 경험."
